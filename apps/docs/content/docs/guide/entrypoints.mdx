title: "Entrypoints"

Entrypoints decide how a Review Task starts. A task can be selected by a change request event, an `@pipr` command comment, or a local CLI command.

## `pipr.review` entrypoints

The quickest setup uses `pipr.review(...)`:

```ts twoslash
// @include: pipr-reviewer
pipr.review({
  id: "review",
  reviewer,
  entrypoints: {
    changeRequest: ["opened", "updated", "reopened", "ready"],
    command: { pattern: "@pipr review", permission: "write" },
    local: "review",
  },
  inlineComments: { max: 5 },
});
```

Set an entrypoint to `false` to disable it:

```ts twoslash
// @include: pipr-reviewer
pipr.review({
  id: "review",
  reviewer,
  entrypoints: {
    changeRequest: ["opened", "updated"],
    command: false,
    local: "review",
  },
});
```

## Change request actions

Supported public action names are:

```text
opened | updated | reopened | ready | closed
```

GitHub native actions are normalized into those provider-neutral names. For example, a pull request synchronize event maps to `updated`.

Use change request triggers for automatic review:

```ts twoslash
// @include: pipr-task
pipr.on.changeRequest({
  actions: ["opened", "updated"],
  task,
});
```

## Commands

Commands run from pull request issue comments and must start with `@pipr`:

```ts twoslash
// @include: pipr-security-task
pipr.command({
  pattern: "@pipr security",
  permission: "write",
  description: "Run the security review task.",
  task: securityTask,
});
```

Command tasks can publish a command response:

```ts twoslash
// @include: pipr-ask-agent
const ask = pipr.task<{ question: string }>({
  name: "ask",
  async run(ctx, input) {
    const manifest = await ctx.change.diffManifest({ compressed: true });
    const answer = await ctx.pi.run(askAgent, {
      question: input.question,
      manifest,
    });

    await ctx.command?.reply(answer.body);
  },
});

pipr.command({
  pattern: "@pipr ask <question...>",
  permission: "read",
  parse: ({ question }) => ({ question }),
  task: ask,
});
```

Command responses are normal pull request issue comments keyed to the source command comment. They do not update the Main Review Comment and do not publish Inline Review Comments.

## Command grammar

Use static words and named captures:

```ts twoslash
const pattern = "@pipr release-note <version>";
```

Use a final rest capture for free-form text:

```ts twoslash
const pattern = "@pipr ask <question...>";
```

The rest capture must be the final required position. This keeps command parsing deterministic.

## Permissions

Permission levels are provider-neutral:

```text
read < triage < write < maintain < admin
```

GitHub repository permissions are normalized into these levels by the GitHub adapter. Command permission defaults to `write`.

Use lower permission only for commands that cannot publish review findings or expose sensitive context:

```ts twoslash
// @include: pipr-ask-task
pipr.command({
  pattern: "@pipr ask <question...>",
  permission: "read",
  task: ask,
});
```

## Local entrypoints

Local entrypoints run without GitHub publication:

```ts twoslash
// @include: pipr-security-task
pipr.local({ name: "security", task: securityTask });
```

Run them with:

```bash
pipr run security --base origin/main
```

The default review shorthand runs the local entrypoint named `review`:

```bash
pipr review --base origin/main
```

See [Local Runs](/docs/guide/local-runs) for base/head behavior.

## Event selection checklist

| Need | Entrypoint |
| --- | --- |
| Automatic review on pull request updates | `changeRequest: ["opened", "updated"]` |
| Manual reviewer rerun | `command: { pattern: "@pipr review" }` |
| Free-form maintainer question | `pipr.command({ pattern: "@pipr ask <question...>" })` |
| Local preflight before pushing | `local: "review"` or `pipr.local(...)` |
| No automatic comments | Disable change request entrypoint and use command or local only |
